群聊功能 — 架构方案对比

基于需求：兴趣群组 / 500-1000人 / 三级管理 / 深度联动 / 实时+离线

需求总结

群组类型

兴趣/话题群组，长期社区

加入方式

群主可选：开放 / 审批 / 邀请制

规模

上限 500-1000 人，管理员体系

消息类型

文字/图片/文件/@提及/回复/投票/活动

管理

群主 → 管理员 → 普通成员

联动

分享卡片 + 群内发任务/活动/组队

选择架构方案

三种方案各有取舍，点击选择你倾向的方向

A

扩展现有 Task Chat 模型

复用现有的 Message 表 + TaskParticipant 模式，新增 ChatGroup 表作为群组元数据，消息仍走 Message 表加 group_id 字段。

优点

  • 复用现有 WebSocket、消息模型、已读追踪
  • 开发量最小，迭代最快
  • 私聊/任务聊/群聊共享同一套基础设施

缺点

  • Message 表会变得很大，混合多种聊天类型
  • 500+ 人群的查询性能可能有瓶颈
  • 群组特有功能（投票、公告）需要硬塞进现有结构
⏱ 预计工作量：中等  |  🏗 架构风险：中
B

独立 Conversation 抽象层（推荐）

新建 Conversation 统一抽象（type: private/task/group），所有聊天类型统一管理。群聊相关新增 GroupInfo、GroupMember、GroupMessage 等独立表。

优点

  • 架构清晰，群聊有独立的表和索引，不污染现有数据
  • 可以为群消息单独优化（分区、归档）
  • 未来扩展容易（频道、直播间等）
  • 群组特有功能（投票、角色、公告）有天然存放位置

缺点

  • 开发量较大，需要迁移或适配现有聊天
  • 需要统一聊天列表的聚合查询
  • WebSocket 路由需要改造
⭐ 推荐  |  ⏱ 工作量：较大  |  🏗 架构风险：低
C

群聊完全独立模块

群聊作为完全独立的微服务/模块，有自己的数据库表、API 路由、WebSocket 连接。与现有聊天系统完全解耦。

优点

  • 零风险：完全不影响现有聊天功能
  • 可以独立部署和扩展
  • 技术选型自由（可以用更适合群聊的方案）

缺点

  • 大量重复代码（WebSocket、消息模型、已读追踪）
  • 用户体验割裂（两套聊天系统）
  • 维护成本翻倍
  • 联动功能实现更复杂
⏱ 工作量：最大  |  🏗 架构风险：低但维护成本高

💡 我的推荐：方案 B

方案 B 在架构清晰度和开发可行性之间取得了最好的平衡。通过引入 Conversation 抽象层，不仅解决群聊需求，还能统一现有的私聊和任务聊天，为将来的扩展打下基础。群聊的特有功能（投票、公告、角色管理）有天然的存放位置，不会污染现有的 Message 表。

虽然工作量比方案 A 大，但可以分阶段实施：先建群聊独立表，现有聊天暂不迁移，后续再统一到 Conversation 模型下。

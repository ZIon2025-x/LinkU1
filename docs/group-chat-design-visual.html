独立 Conversation 抽象层 — 兴趣群组 / 500-1000人 / 三级管理 / 深度联动

🏗 架构方案：独立群聊表 + 未来 Conversation 统一

分阶段策略

第一阶段（本次） 只建群聊相关的新表（chat_groups、group_messages 等），现有私聊/任务聊天保持不动。

第二阶段（后续） 引入 Conversation 统一抽象，把私聊/任务聊天/群聊合并到一个聊天列表模型下。

优点

  • 架构清晰，群聊有独立表和索引
  • 不污染现有 Message 表
  • 群组特有功能有天然存放位置
  • 可为群消息单独优化（分区、归档）
  • 未来扩展容易（频道、直播间等）

代价

  • 开发量较大
  • 需要在聊天列表聚合查询
  • WebSocket 路由需改造

关键设计决策

✅ group_messages 独立表

不与现有 Message 表混用，避免巨表问题。独立索引 (group_id, created_at DESC) 保证群聊查询性能。

✅ 游标分页（cursor-based）

复用任务聊天模式：基于 last_read_message_id，适合消息流无限滚动。

✅ 已读追踪：每用户每群一条记录

复用 MessageReadCursor 模式，不为每条消息标 read，1000 人群也只有 1000 条已读记录。

✅ mentions 用 JSON 数组

["user_id_1", "user_id_2", "all"] — 简单高效，支持 @全体，前端解析渲染高亮。

✅ meta 字段灵活存储

分享的任务/商品/活动卡片存在 meta JSON，不同 message_type 对应不同 schema。

✅ 软删除 + 状态管理

成员用 status (active/banned/left)，群组用 is_active，不做物理删除。

索引策略

-- 消息查询（群聊主查询路径）
CREATE INDEX ix_gmsg_group_created ON group_messages(group_id, created_at DESC);
CREATE INDEX ix_gmsg_group_pinned ON group_messages(group_id) WHERE is_pinned = true;

-- 成员查询
CREATE INDEX ix_gmember_user ON group_members(user_id) WHERE status = 'active';
CREATE INDEX ix_gmember_group ON group_members(group_id, role);

-- 已读追踪
CREATE UNIQUE INDEX ux_group_reads ON group_message_reads(group_id, user_id);

-- 群组发现
CREATE INDEX ix_groups_category ON chat_groups(category) WHERE is_active = true;
CREATE INDEX ix_groups_tags ON chat_groups USING gin(tags);

⚡ WebSocket 设计

── 现有连接复用 ──

用户的 WebSocket 连接不变：wss://{host}/ws/chat/{userId}
群聊消息通过 group_id 字段区分，不新建连接。

── 新增消息类型 ──

OUTBOUND (客户端 → 服务器):
├── group_message:  { type, group_id, content, msg_type, reply_to?, mentions? }
├── group_typing:   { type, group_id }
└── group_read:     { type, group_id, last_read_message_id }

INBOUND (服务器 → 客户端):
├── group_message:  { type, group_id, message: {...} }
├── group_typing:   { type, group_id, sender_id, sender_name }
├── group_read:     { type, group_id, user_id, last_read_message_id }
├── group_member_joined:  { type, group_id, user: {...} }
├── group_member_left:    { type, group_id, user_id }
├── group_updated:        { type, group_id, changes: {...} }
└── group_poll_vote:      { type, group_id, poll_id, voter_id, option }

── 广播策略 ──

发送群消息时：
  1. 持久化到 group_messages 表
  2. 查询该群所有 active 成员的 user_id
  3. 批量推送：遍历在线成员，通过 WebSocketManager.send_to_user() 逐一发送
  4. 离线推送：不在线的成员 → 加入 FCM/APNs 推送队列 (Celery task)
  5. 更新群组的 last_message 缓存 (Redis)

── 性能优化 ──

• 群成员列表缓存在 Redis SET: group_members:{group_id}（避免每条消息都查 DB）
• typing 事件只广播给最近 5 分钟活跃的成员（减少无效推送）
• 大群（>200人）的 typing 事件降频到每 5 秒最多一次

BLoC 层级

Page 级 — GroupListBloc

挂在群组列表页。管理"我的群组"和"发现群组"两个 tab 的状态，支持搜索/筛选/分页。

Page 级 — GroupChatBloc

挂在群聊页。管理消息列表、发送、WebSocket 订阅、typing 指示器、@提及解析。复用 ChatBloc 的乐观更新和游标分页模式。

Page 级 — GroupManageBloc

挂在群组信息/管理页。处理成员列表、角色变更、禁言、踢出、加入审批、群设置编辑。

复用 — NotificationBloc (Root)

已有的根级 NotificationBloc 扩展：群聊未读计数、新消息通知、@提及通知。

本地缓存策略 (Hive)

群组列表缓存

Box: group_list — 缓存我的群组 + 最后消息 + 未读数。打开 app 先显示缓存，后台刷新。

消息缓存

Box: group_messages_{groupId} — 每个群一个 box，缓存最近 200 条。超过的自动淘汰最旧的。

成员缓存

Box: group_members_{groupId} — 缓存成员列表，TTL 5 分钟。

📅 分阶段实施

Phase 1 — 核心群聊 MVP
  • 数据库表 + 迁移脚本
  • 群组 CRUD API
  • 成员加入/退出/邀请
  • 群消息发送/接收 (文字+图片)
  • WebSocket 群消息广播
  • Flutter: 群列表 + 群聊页
  • Hive 本地缓存
Phase 2 — 管理 & 互动
  • 三级角色管理
  • 禁言/踢出/审批加入
  • @提及 + 消息回复/引用
  • 公告置顶
  • 群组发现（搜索/分类/标签）
  • 文件分享
Phase 3 — 高级功能
  • 投票功能
  • 深度联动（分享任务/商品/活动卡片）
  • 群内发起任务/活动
  • FCM/APNs 离线推送
Phase 4 — 统一 & 优化
  • Conversation 抽象层统一聊天列表
  • 消息归档策略（冷热分离）
  • 大群性能优化
  • 消息搜索
💡 设计原则： 每个 Phase 都是可独立发布的。Phase 1 完成后用户就能创建群、聊天；后续 Phase 逐步丰富功能。 不需要等所有功能做完才上线。
